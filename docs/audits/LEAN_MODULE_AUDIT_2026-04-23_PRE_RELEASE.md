# Orbcrypt Pre-Release Comprehensive Lean Module Audit

**Date:** 2026-04-23
**Auditor:** Claude (automated Lean module audit)
**Scope:** Every `.lean` source file under `Orbcrypt/`, the root `Orbcrypt.lean`
aggregator, `lakefile.lean`, `lean-toolchain`, audit scripts, and the architectural
coherence of the project as a whole.
**Branch:** `claude/audit-lean-modules-gYcbJ`
**Project version:** 0.1.6 (Workstream L bump)
**Baseline:** post-Workstream-N (finding I1 + I5 landing), 38 modules,
347 public declarations.

## 0. Meta-Note on Audit Methodology

This audit is intentionally harsher than the 2026-04-14, 2026-04-18, and
2026-04-21 predecessor audits because the project is approaching its first
major-version release. Every line of Lean has been read directly; none of
the findings below are generated by trusting the module docstring / CLAUDE.md
snapshot text / VERIFICATION_REPORT.md tables alone. Where the
documentation *does* track the code accurately, that fact is stated
explicitly; where the documentation diverges from the code, the divergence
is reported as a separate documentation finding.

A finding is **blocking** if it could cause a consumer of the v1.0 release
to form a false belief about what has been proved. All other findings are
either **enhancement** (nice-to-have before v1.0), **research** (requires
open research follow-up), or **cosmetic** (naming / whitespace / comment
hygiene).

Severity ordering: **CRITICAL → HIGH → MEDIUM → LOW → INFO**. A CRITICAL
finding blocks the v1.0 tag outright (e.g. an unsoundness, a completely
false advertised claim, a security vulnerability in a cryptographic
primitive). A HIGH finding should block the v1.0 tag unless explicitly
accepted by the project owner with a tracking item. MEDIUM and below are
polish.

## 3. Module-by-Module Findings

### 3.1 `Orbcrypt/GroupAction/Basic.lean` (118 lines)

Reviewed in full.

**Content.** Three sections mirroring plan work units 2.1–2.4:
- `orbit`, `stabilizer` abbrevs over `MulAction.orbit` / `MulAction.stabilizer`.
- `orbit_disjoint_or_eq`: any two orbits are equal or disjoint, via a
  `by_cases` on `x ∈ MulAction.orbit G y` using `MulAction.orbit_eq_iff`.
- `orbit_stabilizer`: wrapper around
  `MulAction.card_orbit_mul_card_stabilizer_eq_card_group`.
- `smul_mem_orbit` and `orbit_eq_of_smul`: wrappers around
  `MulAction.mem_orbit` and `MulAction.orbit_smul`.

**Assessment.** Clean, well-documented, zero `sorry`. The `abbrev`s
`orbit` / `stabilizer` introduce name-shadowing with Mathlib but that
is intentional and the project imports `Orbcrypt.GroupAction.Basic`
into a namespace so clashes are contained.

**Finding B-01 (LOW, cosmetic).** The `orbit_disjoint_or_eq` proof at
lines 65–71 uses a somewhat convoluted sequence of `.mp`/`.mpr` calls.
Mathlib has `MulAction.orbitRel` with an equivalence relation; a
one-liner via `Setoid.equivalence_classes_disjoint_or_eq` (or similar)
might be cleaner. Not a correctness concern.

**Finding B-02 (INFO).** The module declares `variable {G : Type*} {X : Type*}`
at the namespace level but each `section` redeclares type class constraints
`[Group G] [MulAction G X]` locally. This is good practice (avoids
autoImplicit leakage) and is consistent with the
`autoImplicit := false` project setting.

### 3.2 `Orbcrypt/GroupAction/Canonical.lean` (111 lines)

Reviewed in full.

**Content.** Defines the `CanonicalForm` structure (canonicalization
function + orbit-membership field + orbit-characterisation field),
proves three forward-direction theorems (`canon_eq_implies_orbit_eq`,
`orbit_eq_implies_canon_eq`, `canon_eq_of_mem_orbit`) and idempotence
(`canon_idem`).

**Assessment.** Mathematically clean. The structure's two fields
`mem_orbit` and `orbit_iff` are the minimal data to witness a canonical
form; every downstream proof extracts them uniformly.

**Finding B-03 (MEDIUM, research).** The `CanonicalForm` structure has
**no computability guarantee** — the `canon` function is an arbitrary
`X → X` obligation. In the current formalization this is fine because
orbit encryption at the abstract level never needs to *run* the canonical
form — decryption is `noncomputable` via `Exists.choose`. However, for
the production GAP / reference implementation (and for any planned
Phase-15 C implementation), the `canon` function *must* be implementable,
typically as lex-minimal orbit element. There is no Lean-level connection
between the abstract `CanonicalForm` structure and any complexity or
computability witness. This is acceptable for the formalization, but a
v1.0 release should explicitly state that `CanonicalForm` admits
non-computable witnesses and that implementations (GAP, C) carry
per-construction complexity arguments separately.

**Finding B-04 (LOW, enhancement).** The structure has no bundled
idempotence field — `canon_idem` is proved as a theorem from
`orbit_iff` + `mem_orbit`. This is idiomatic but means consumers who
need idempotence must invoke `canon_idem` explicitly rather than
`can.canon_idem`. Not blocking.

### 3.3 `Orbcrypt/GroupAction/Invariant.lean` (155 lines)

Reviewed in full.

**Content.**
- `IsGInvariant f`: `∀ g x, f (g • x) = f x`.
- Composition closure (`IsGInvariant.comp`), constants
  (`isGInvariant_const`).
- `invariant_const_on_orbit`: G-invariant functions are constant on orbits.
- `IsSeparating f x₀ x₁`: invariance + `f x₀ ≠ f x₁`.
- `separating_implies_distinct_orbits`.
- `canonical_isGInvariant`: canonical form is G-invariant.

**Assessment.** Correct and clean. The key lemma `invariant_const_on_orbit`
is the tip of the spear for the invariant attack theorem; the proof
destructures orbit membership via `MulAction.mem_orbit_iff.mp` and
rewrites by the invariance hypothesis in one step.

**Finding B-05 (LOW, enhancement).** `isGInvariant_const` is the only
closure lemma beyond `comp`. Other useful closures — sums,
min/max under a fintype, pointwise equality propagation — are absent
from this module but are used repeatedly in downstream constructions
(Hamming weight, canonical form G-invariance, etc.). This is fine but
a v1.0 polish pass could collect the canonical closure properties in
one place.

**Finding B-06 (MEDIUM, research).** `IsSeparating` is defined for a
pair `(x₀, x₁)` only — a *pointwise* separating condition. The
literature on invariant-based distinguishers frequently parameterises
by a **distribution** over a message space, which is a quantitatively
stronger notion. The downstream `invariant_attack` theorem uses the
pointwise form and achieves advantage 1/2 (complete break); it does
not speak about sub-complete distinguishers. For a v1.0 release this
is fine because complete break is the security-relevant outcome; but
any future *partial-break* analysis (e.g. showing adversary advantage ≤ ε
for some specific invariant over a skewed message distribution) would
need to refactor `IsSeparating` to carry the distribution as a parameter.
Tracked as research item R-01.

### 3.4 `Orbcrypt/Crypto/Scheme.lean` (112 lines)

Reviewed in full.

**Content.** `OrbitEncScheme` structure bundling `reps : M → X`,
`reps_distinct`, and `canonForm : CanonicalForm G X`. `encrypt g m = g • reps m`.
`decrypt c` uses classical `Exists.choose` to pick a matching message or
`none`.

**Finding C-01 (MEDIUM, enhancement).** `decrypt` is `noncomputable`
because of `Exists.choose`. This is *formally* fine — `correctness`
proves `decrypt (encrypt g m) = some m` for honest ciphertexts — but
the formalization does not include a *computable* `Decidable`
decryption algorithm. Production consumers (GAP prototype, planned C
implementation) cannot point to a machine-checked `def` that mirrors
their implementation. Recommendation for v1.0 polish: add a
`decryptCompute` variant using `Finset.decidableExistsOfFinset` + a
`[DecidableEq X]` search; prove
`decrypt = decryptCompute` under `Classical.choose`-compatible
preconditions. Not blocking release, but a v1.0 completeness gap.

**Finding C-02 (LOW, cosmetic).** `encrypt` has no inverse-direction
simp lemma. Consumers who want to rewrite an encrypted message back
to `g • reps m` must `unfold encrypt` each time. Adding `@[simp]
theorem encrypt_eq : encrypt scheme g m = g • scheme.reps m := rfl`
would smooth downstream proofs.

**Finding C-03 (LOW, design).** The docstring says "`M` is left
abstract… `Fintype M` is added only to `decrypt`." But the
`reps_distinct` field quantifies over `M × M` directly without a
`Fintype` constraint. This is consistent with the code but worth an
explicit *"why"* comment: because `reps_distinct` is a `Prop` obligation
and does not require enumeration, a finite `M` is only required for
the decryption search. Fine as-is; polishing would just be a
docstring addition.

### 3.5 `Orbcrypt/Crypto/Security.lean` (232 lines)

Reviewed in full.

**Content.** `Adversary` structure with deterministic `choose` /
`guess`; `hasAdvantage`, `IsSecure`; `hasAdvantageDistinct`,
`IsSecureDistinct`; `isSecure_implies_isSecureDistinct`
(one-line proof);`hasAdvantageDistinct_iff` (`Iff.rfl`).

**Finding C-04 (MEDIUM, design).** Module docstring
notes "the adversary does not know the secret group `G`", but the
adversary *does* receive `scheme.reps : M → X` as input. In Orbcrypt,
knowing `reps` means knowing the orbit representatives, which — under
the original AOE design where `reps` is a *secret* public key — is
fine; but the docstring later reads "orbit representatives as public
parameters". These two framings are in tension: the adversary receives
representatives because the scheme *publishes* them (or they are derivable
from a public parameter set), but the post-KEM architecture (Phase 7)
hides them behind a single `basePoint`. This module was written pre-KEM
and the wording should be updated to reflect the fact that the
scheme-level game always hands the full `reps` map to the adversary —
which is the strictly stronger setting, not a "does not know the
secret" framing. No correctness impact; documentation-only.

**Finding C-05 (HIGH, release messaging).**
`isSecure_implies_isSecureDistinct` proves the *stronger* game
(`IsSecure`, which accepts collision-choice `(m, m)`) implies the
classical IND-1-CPA game. The docstring at lines 200–206 says
"the converse is false in general" but gives no counterexample. A v1.0
release should either (a) give an explicit counter-example witness
(an adversary that is secure under `IsSecureDistinct` but breaks
`IsSecure` by exploiting collision choices), or (b) cite the collision
case via the formal `Adversary` that always returns `(m, m)` with a
`guess` that leaks information — this can be constructed explicitly.
The asymmetry is not blocking but a pre-release explicit witness would
remove ambiguity for external reviewers.

**Finding C-06 (MEDIUM, correctness).** Notice that `hasAdvantage` does
*not* re-sample the group element challenge across `g₀, g₁`. A
conventional probabilistic IND-1-CPA game samples a *single* random
group element for the challenge and a random challenge bit, then the
adversary guesses the bit. The deterministic `hasAdvantage` uses
`∃ g₀ g₁`, which is a strictly *weaker* definition of distinguishability
than the two-distribution advantage used in `indCPAAdvantage` (which
averages over uniform `g`). The formal bridge
`concrete_oia_implies_1cpa` works despite this mismatch because
`ConcreteOIA` bounds the stronger two-distribution advantage; but the
*deterministic* `IsSecure` and the *probabilistic* `indCPAAdvantage`
are not mutually characterising. This is documented obliquely in the
scaffolding vs quantitative distinction (Workstream J) but the fact
that the two games are genuinely different — not merely deterministic
vs probabilistic renderings of the same game — deserves a dedicated
release note. Not blocking; recommend `docs/VERIFICATION_REPORT.md`
release-readiness prose clarify this.

### 3.6 `Orbcrypt/Crypto/OIA.lean` (201 lines)

Reviewed in full. The file contains an extensive ~140-line docstring
discussion of the OIA, followed by a 5-line `def` and a 12-line
comment pointer back to the docstring.

**Content.** `OIA scheme` is defined as
`∀ (f : X → Bool) (m₀ m₁ : M) (g₀ g₁ : G), f (g₀ • reps m₀) = f (g₁ • reps m₁)`.
The docstring explicitly discloses the scaffolding / vacuity status.

**Finding C-07 (HIGH, release messaging).** The OIA docstring at lines
46–67 says the deterministic OIA is `False` for any non-trivial scheme
"because `f := fun x => decide (x = reps m₀)` is a distinguisher". This
claim is stated in prose but **not proved formally in Lean**. Before a
v1.0 tag, this should be a *named theorem* — e.g.
`det_oia_false_of_distinct : scheme.reps m₀ ≠ scheme.reps m₁ → ¬ OIA scheme`
— so that downstream readers can machine-check the vacuity claim
rather than trusting prose. The proof is one-liner via the membership
distinguisher `decide (x = scheme.reps m₀) = true` at `m₀`, `false`
at `m₁` (when the representatives are distinct). Without this, the
scaffolding disclosure in Workstream J is not itself machine-checked.

**Finding C-08 (MEDIUM, design).** The OIA quantifies over `(g₀, g₁)`
*independently*, and over `f` universally. This is strictly stronger
than the two-distribution advantage formulation. The docstring
acknowledges this but does not formally relate the deterministic OIA
to the `concrete_oia_implies_1cpa` bound — in fact, the
`det_oia_implies_concrete_zero` bridge in `Crypto/CompOIA.lean` proves
the scheme-level connection. The module-docstring relationship
explanation at lines 70–83 would be strengthened by a direct
backlink to `det_oia_implies_concrete_zero` in a "See also" section.

**Finding C-09 (LOW, cosmetic).** The namespace `Orbcrypt`
re-implements documentation pointers in lines 109–119 via a comment
block that merely echoes the top-of-file docstring. This is a
documentation duplication noted already by audit plans. Harmless but
could be deleted in a v1.0 cleanup.

### 3.7 `Orbcrypt/Crypto/CompOIA.lean` (244 lines)

Reviewed in full.

**Content.** `orbitDist (x : X) := PMF.map (· • x) (uniformPMF G)`;
support lemmas; `ConcreteOIA scheme ε` as a universal bound over
`(D, m₀, m₁)` of the orbit-distribution advantage; closure lemmas
(`_mono`, `_one`, `_zero_implies_perfect`); `SchemeFamily` indexed by
security parameter; `CompOIA sf`; `det_oia_implies_concrete_zero`
bridge theorem.

**Finding C-10 (MEDIUM, research).** `det_oia_implies_concrete_zero`
proves the deterministic → probabilistic direction at `ε = 0`. There
is *no corresponding* `probabilistic → deterministic at ε = 0`
direction — because at `ε = 0`, `ConcreteOIA scheme 0` is *less*
demanding than `OIA` (the probabilistic form only quantifies over
pairs of same-G/same-sampling distributions, not over all `(g₀, g₁)`
combinations). This asymmetry is noted in the CLAUDE.md "Vacuity map"
but should also be formally captured as a `¬` theorem or an explicit
counterexample. Not blocking.

**Finding C-11 (LOW, design).** `SchemeFamily` uses four separate
typeclass-instance fields (`instGroup`, `instAction`, `instFintype`,
`instNonempty`, `instDecEq`) rather than bundling them via synthetic
instances. This is correct Lean but slightly heavier than a single
"instances" bundle. The downstream helpers (`repsAt`, `orbitDistAt`,
`advantageAt`) then have to thread the five fields manually via
`@`-syntax. This is the pain that F-13 tried to alleviate; the
helpers work but the underlying design choice is debt.

**Finding C-12 (MEDIUM, research).** The asymptotic `CompOIA` /
`CompIsSecure` framework uses `IsNegligible : (ℕ → ℝ) → Prop`, but
never formally connects this to PPT (polynomial-time) adversary
bounds. The standard definition requires
`∃ c, ∀ polynomial `p`, advantage n ≤ 1/p(n)` for all "PPT" adversaries,
whereas Orbcrypt's `IsNegligible` bounds advantage by
`∀ c, ∃ n₀, ∀ n ≥ n₀, |f n| < n⁻ᶜ`. The two are *classically*
equivalent but the formalization does not bind adversaries to a
complexity class — `CompIsSecure` quantifies over ALL adversaries,
not just PPT ones. For a v1.0 release the distinction is worth an
explicit "information-theoretic vs computational" note — the current
formalization proves the former but the term "Computational OIA"
suggests the latter.

### 3.8 `Orbcrypt/Crypto/CompSecurity.lean` (482 lines)

Reviewed in full.

**Content.** Probabilistic IND-1-CPA advantage (`indCPAAdvantage`);
`concrete_oia_implies_1cpa` (the main non-vacuous theorem);
`concreteOIA_one_meaningful`; `CompIsSecure`; `comp_oia_implies_1cpa`;
`single_query_bound`; `DistinctMultiQueryAdversary`; `perQueryAdvantage`
with nonneg/le_one/bound lemmas; `hybridDist`; `indQCPAAdvantage`;
`indQCPA_bound_via_hybrid` (E8c, telescoping); `indQCPA_bound_recovers_single_query`;
`indCPAAdvantage_collision_zero` (K4).

**Finding C-13 (HIGH, blocking for v1.0).** `indQCPA_bound_via_hybrid`
carries the per-step bound `h_step` as an **explicit hypothesis**, but
it is documented as the "multi-query IND-Q-CPA theorem from
ConcreteOIA" in external documentation (VERIFICATION_REPORT). The
hypothesis `h_step` is in general **hard** to discharge from
`ConcreteOIA` alone — it requires a marginal-independence argument
over `uniformPMFTuple`, which is not provided in Orbcrypt. Without a
*formal* proof that `h_step` is discharge-able from `ConcreteOIA`, the
theorem is effectively "*if* you can bound adjacent hybrids then Q·ε
follows", which is mathematically correct but has limited value. This
is disclosed in CLAUDE.md (Workstream E8b deferred) but external
summaries like "multi-query IND-Q-CPA under ConcreteOIA" overstate
the current state. Before v1.0, EITHER (a) discharge E8b formally, OR
(b) rename `indQCPA_bound_via_hybrid` to make it clear the h_step
hypothesis is a real gap (e.g. `indQCPA_from_perStepBound`).

**Finding C-14 (MEDIUM, correctness).** `hybridDist` at lines 387–394
uses `PMF.map` over `uniformPMFTuple G Q`, building the ciphertext
tuple by `if j.val < i then ... else ...`. The boundary convention
is: `i = 0` → all coordinates are *right* messages; `i = Q` → all
coordinates are *left*. `indQCPAAdvantage` at line 408 then takes
`advantage (A.guess) (hybridDist … Q) (hybridDist … 0)`, matching
"all-left minus all-right". This is consistent but the "left/right"
wording in the docstring at lines 369–380 is reversed relative to
the convention used in `indCPAAdvantage` where `orbitDist (reps m₀)`
is *first* and `orbitDist (reps m₁)` is *second*. Small; worth a
correction for readability.

**Finding C-15 (LOW, cosmetic).** `concreteOIA_one_meaningful` at
line 197–201 is named misleadingly — it just proves `indCPAAdvantage ≤
1`, which does not "demonstrate the ConcreteOIA(1) predicate is
satisfiable" in the sense of satisfiability of ConcreteOIA. The
correct name would be `indCPAAdvantage_le_one`; the satisfiability of
`ConcreteOIA 1` is actually proved by `concreteOIA_one`. The naming
could mislead consumers expecting a logical-existence witness. Not
blocking.

**Finding C-16 (HIGH, research gap).** `indCPAAdvantage_collision_zero`
discharges the collision branch via `advantage_self`. While this
removes the need for a `_distinct` probabilistic corollary, it does
not *eliminate* the game asymmetry — an adversary that *always*
returns `(m, m)` has advantage 0 (trivially secure) under
`concrete_oia_implies_1cpa`, but might not be what a consumer
expecting IND-1-CPA security against reasonable adversaries actually
cares about. The real concern is adversaries that make a collision
"sometimes" and a distinct challenge "other times"; the collision
branch and the distinct branch both satisfy the bound but they are
not the same game. This is a subtle point that the release messaging
should clarify.

### 3.9 `Orbcrypt/Theorems/Correctness.lean` (137 lines)

Reviewed in full.

**Content.** `encrypt_mem_orbit`; `canon_encrypt` (via
`canonical_isGInvariant`); `decrypt_unique`; `correctness` theorem.

**Finding D-01 (LOW, enhancement).** `correctness` uses
`h_exists.choose` then proves `h_exists.choose = m` via
`decrypt_unique`. This is elegant but could be phrased more directly:
build the unique witness via `Fintype.exists_unique` (proved from the
`reps_distinct` injectivity) to get a computable (via `Classical`)
unique choice. Not blocking.

**Finding D-02 (MEDIUM, research).** `correctness` is a *deterministic
exact-match* correctness — it shows the decryption output is `some m`.
A probabilistic correctness theorem (e.g. `Pr[decrypt(encrypt g m) = some m] = 1`)
is trivially equivalent *in this formalization* (because encrypt and
decrypt are both deterministic), but a release-quality formalization
should document that the "1-probability correctness" matches DEVELOPMENT.md's
§4.2 claim exactly. Currently this is only implied by the lack of
probabilistic decryption. Not blocking.

**Finding D-03 (LOW, cosmetic).** `decrypt_unique` uses a `by_contra
h_ne` followed by `exact absurd h_orbits (scheme.reps_distinct m' m h_ne)`.
This could use `exact scheme.reps_distinct m' m h_ne h_orbits |>.elim`
to avoid the lemma-name overhead. Cosmetic.

### 3.10 `Orbcrypt/Theorems/InvariantAttack.lean` (174 lines)

Reviewed in full.

**Content.** `invariantAttackAdversary` construction with
simp lemmas; `invariant_on_encrypt` helper; `invariantAttackAdversary_correct`
case-split proof; `invariant_attack` assembly.

**Finding D-04 (HIGH, release messaging).** The `invariant_attack`
docstring (lines 122–155) discusses advantage conventions at length
but the actual theorem statement is just `∃ A, hasAdvantage scheme A`
— i.e. *existence of an adversary with at least one distinguishing
`(g₀, g₁)` pair*. This is weaker than "advantage 1/2" or "complete
break" as rendered in CLAUDE.md's headline theorem table. The gap
between "distinguishes on one (g₀, g₁) pair" and "always
distinguishes" (which is what the probabilistic advantage-1 claim
requires) is significant. The proof exhibits `g₀ = g₁ = 1` so the
adversary *does* distinguish perfectly on that pair, but the theorem
statement does not capture this strongly. For v1.0 release messaging,
either upgrade the theorem to state "`∃ A, ∀ g₀ g₁, hasDistinguishingBehavior
A g₀ g₁`" or tone down external claims to match the current statement.

**Finding D-05 (MEDIUM, research).** The attack uses `g₀ = g₁ = 1`.
A subtle issue: if the scheme's `reps_distinct` did not hold, this
attack would still "succeed" (produce distinct guesses) but the
scheme would also be ill-formed. The theorem does not explicitly
use `reps_distinct`, which means it would "break" a scheme where
`reps m₀ = reps m₁`. In practice `reps_distinct` is a scheme
invariant, so this is vacuous, but the minimal-assumptions structure
of the proof means the theorem is stated without `reps_distinct`,
which could mislead a reader into thinking the attack is "harder" than
it actually is. Not blocking; recommend a docstring note that the

Reviewed in full.

`keyDerive := kd` — passed directly as an argument. A cleaner design

### 3.24 `Orbcrypt/KeyMgmt/SeedKey.lean` (365 lines)

Reviewed in full.

**Content.** `SeedKey` structure with `compression` bit-length witness
(Workstream L1); `seed_kem_correctness`; `HGOEKeyExpansion`;
`seed_determines_key`, `seed_determines_canon`;
`OrbitEncScheme.toSeedKey` bridge with `hGroupNontrivial`.

**Finding H-01 (HIGH, research).** The `compression` field uses
`Nat.log 2` which rounds down. So `Nat.log 2 1 = 0`, `Nat.log 2 2 = 1`,
`Nat.log 2 3 = 1` (rounding down). Consequently `compression` at

what matters. The Lean-level guarantee **does not enforce a
quantitative compression ratio**; it only enforces strict inequality
by ≥ 1 bit. A v1.0 claim of "256-bit seed compresses a 10-trillion-bit
group" should not be read from the Lean `compression` field alone —
the field only certifies "1 fewer bit". Recommend either upgrading

than "meaningful compression".

Seed → CanonicalForm G X` — a *function* from seeds to canonical
forms. But the structure only uses this function applied to `seed`
(the structure's own field). Having `expand` as a field makes the
structure conceptually "bigger" than it needs to be. A simpler
design: `canonForm : CanonicalForm G X` (precomputed) + `sampleGroup`
function. The current design anticipates consumers iterating
`expand` over multiple seeds, which is not currently supported
elsewhere. Not blocking; refactor opportunity.

**Finding H-03 (MEDIUM, research).** `HGOEKeyExpansion` is a
*specification* (Prop-valued fields), not an algorithm. `group_large_enough :
group_order_log ≥ 128` hard-codes λ = 128 bits. For λ ∈ {80, 192, 256}
(the other security levels documented in DEVELOPMENT.md), this field
would fail. Should be parameterized by λ. **This is a blocker for
matching the documented multi-level parameter story** — you cannot
instantiate `HGOEKeyExpansion` for λ = 256 because the hypothesis
`group_order_log ≥ 128` is tied to λ = 128.

**Finding H-04 (LOW, cosmetic).** `HGOEKeyExpansion.reps` has type
`M → Bitstring n` for a message type `M` that is the structure's type
parameter. `M` is declared as the second argument. This is fine but
means the structure is not polymorphic in the underlying bitstring
length independently of the message space.

`keyDerive_canon_eq_of_mem_orbit` (private);
`authEncrypt_is_int_ctxt`.

**Finding I-03 (HIGH, blocking for v1.0).** `authEncrypt_is_int_ctxt`
takes `hOrbitCover : ∀ c : X, c ∈ MulAction.orbit G akem.kem.basePoint`
— the *orbit-cover* hypothesis. In real-world implementations, `X` is
a bitstring space `Bitstring n` and the orbit of any non-trivial
subgroup `G ≤ S_n` is **strictly smaller** than `X` (|orbit| = |G|/|Stab|
by orbit-stabilizer, and `|X| = 2^n`). So `hOrbitCover` is generally
**false** for practical HGOE instantiations. This means INT_CTXT as
formalized *is vacuously provable* (the hypothesis is unfulfillable on
production schemes) but does not apply to the actual production KEM.
This is disclosed in the module docstring as "the intended model" but
*not* clearly in CLAUDE.md's Status column. The theorem should be
classified as **Scaffolding** rather than **Standalone** — it requires
a hypothesis that production KEMs cannot discharge. Release-messaging
critical: the headline claim "INT-CTXT holds for AuthOrbitKEM" needs
to be qualified.

**Finding I-04 (MEDIUM, research).** The private helper
`keyDerive_canon_eq_of_mem_orbit` is the real structural content —
it uses `canon_eq_of_mem_orbit` to prove the key depends only on the
orbit. This does not need the orbit-cover hypothesis. But without
`hOrbitCover`, the INT_CTXT claim cannot bridge from "verify
accepted" to "there's a matching encapsulation"; that step **does**
need orbit-cover. A release-quality alternative is to redefine
`INT_CTXT` to **restrict to ciphertexts in the basepoint's orbit** —
which would both be provable unconditionally AND more aligned with
real-world threat models (attackers producing arbitrary `c ∉ orbit`
are rejected by a pre-check).

**Finding I-05 (MEDIUM, correctness).** `authEncaps` has type

## 6. Cross-Cutting Architectural Findings

### 6.1 The Two Chains Problem

**Finding X-01 (CRITICAL, release messaging).** The project carries
two parallel security-reduction chains:
1. **Deterministic chain** (Phases 3/4/7/10/12): `OIA` / `KEMOIA` /
   `HardnessChain` / `TensorOIA` / `CEOIA` / `GIOIA` — **vacuous on
   every non-trivial scheme**.
2. **Probabilistic chain** (Phase 8 + Workstreams E/G/H):
   `ConcreteOIA` / `ConcreteKEMOIA_uniform` / `ConcreteHardnessChain`
   / `*_viaEncoding` Props — **inhabited only at ε = 1 (trivially
   true)**.

**No scheme instance in the formalization achieves ε < 1 security.**
All "security" theorems are therefore either vacuous (chain 1) or
trivially true (chain 2 at ε = 1). This is correctly disclosed in
the Workstream J snapshot, but the formalization's public API would
be improved by:
- a `def isVacuous (scheme) : Prop := ¬ OIA scheme` witness theorem;
- a policy statement in CLAUDE.md that ε = 1 inhabitants are
  satisfiability witnesses only, never release-messaging targets;
- a research-scope milestone in docs/planning/ tracking concrete
  ε < 1 discharges.

### 6.2 The Non-Computable Decryption

**Finding X-02 (MEDIUM, release-quality).** `decrypt` (scheme level)
and all deterministic-OIA security theorems depend on
`Exists.choose`/`Classical.choice`. While formally sound, the
formalization cannot *execute* decryption. This is acceptable for
a *verification* project, but for a release claiming "formal
verification of a cryptosystem", consumers may expect a computable
reference decapsulation. The GAP implementation provides this but
is not Lean-verified. A v1.0 polish: add `decryptCompute` using
`Finset.choose`/`Fintype.choose` and prove it agrees with `decrypt`
under honest ciphertexts.

### 6.3 Module Layering

**Finding X-03 (LOW, hygiene).** Module dependency graph:
- GroupAction forms the foundation (no downstream dependencies).
- Crypto layer on top of GroupAction.
- Theorems on top of Crypto.
- KEM on top of Crypto + GroupAction.
- Construction on top of GroupAction + Crypto + Theorems.
- Probability (independent).
- KeyMgmt on top of KEM + Construction.
- AEAD on top of KEM + GroupAction.
- Hardness on top of Probability + Crypto.
- PublicKey on top of KEM + GroupAction + Crypto.
- Optimization on top of GroupAction + KEM + Construction.

This layering is clean. Some cross-cutting concerns (e.g., audit
notes referencing OIA appear in many files) are documentation, not
import graph. No cycles.

### 6.4 Universe Polymorphism Posture

**Finding X-04 (LOW, enhancement).** Post-Workstream-M1,
`SurrogateTensor.{u}` is universe-polymorphic. Most other
structures use `Type*` (arbitrary). `SchemeFamily` (post-B2) uses
explicit `universe u v w`. Overall posture is good but not
uniform. A v1.0 polish would audit every structure for consistent
universe handling.

### 6.5 Proof-Computational Complexity

**Finding X-05 (INFO).** The build is ~3,367 jobs (per CLAUDE.md
Workstream M snapshot). This includes Mathlib transitives.
`lake build Orbcrypt` reportedly takes tens of minutes on a cold
cache. Cached builds are much faster. CI relies on Mathlib cache.

## 8. Axiom-Transparency Deep Check

**Finding P-01 (INFO).** All 342 declarations exercised by
`scripts/audit_phase_16.lean` depend only on standard Lean axioms
(`propext`, `Classical.choice`, `Quot.sound`). 133 depend on no
axioms at all. **Zero custom axioms**. Zero `sorryAx`. This
posture is rigorously verified by CI.

**Finding P-02 (INFO).** OIA, KEMOIA, CEOIA, GIOIA, TensorOIA,
CompOIA, ConcreteOIA, and all related predicates are
`Prop`-valued `def`s, not `axiom`s. This is correct and
explicitly avoids the universal-axiom inconsistency trap.

**Finding P-03 (INFO).** CI's axiom-detection hook is robust. The
Perl regex de-wraps multi-line axiom lists (N5, 2026-04-23). Also
validated in CI via `grep -Prn "^axiom\s+..."` for custom axiom
declarations.

**Finding P-04 (LOW).** `CommGroupAction.comm` is a *typeclass
field* (not a Lean `axiom`). Each concrete instance discharges it
with a proof, but the field itself is ∀-quantified over every
instance. This is standard Lean practice, documented, and not a
concern.

**Finding P-05 (MEDIUM, research).** The axiom-transparency check
verifies that declarations depend only on Lean's standard trio. But
it does NOT verify:
- That any `Prop` hypothesis is *satisfiable* on a non-trivial
  scheme. Vacuous hypotheses pass the axiom check.
- That the declarations form a *useful* API. A declaration can
  be axiom-free yet useless (e.g., a theorem about the empty set).

The audit script's non-vacuity witness block (§ 12 of
`audit_phase_16.lean`) addresses the first concern partially, but
only at ε = 1. A v1.0 polish would extend non-vacuity witnesses
to intermediate ε for at least one class of surrogate / encoder.

## 12. Performance & Implementation Deferred Work

These are non-security items needed to make the project practical.

**Z-01.** Phase 15 (Decryption Optimisation in C/C++) is *planned*
but not executed. The GAP implementation runs at ~244 ms/decaps at
λ = 128, which is 4–5 orders of magnitude slower than AES-GCM. A
v1.0 release without Phase 15 delivery ships a formally verified
construction that is not practically deployable.

**Z-02.** The GAP implementation does not have a Lean-verified
counterpart. Every Lean-side guarantee is *about the abstract
specification*; the GAP code's correctness is verified only by
the test suite (13/13 pass).

**Z-03.** No Lean-side formalization of PRF (pseudorandom function)
instances. `SeedKey.sampleGroup` requires a PRF; the formalization
admits any function. For a concrete deployment, a cryptographic
PRF (AES-CTR-DRBG or similar) must be instantiated and assumed
secure.

**Z-04.** No Lean-side formalization of cryptographic hash for
`keyDerive`. Same gap as Z-03 but for the KEM layer.

**Z-05.** No Lean-side formalization of DEM security. `DEM` is a
black box. Production deployment uses AES-GCM.

**Z-06.** The λ = 80, 192, 256 parameter sweeps (Phase 14) are in
GAP benchmarks, but `HGOEKeyExpansion` hard-codes λ = 128 in its
`group_large_enough` field. The formalization does not cover the
other security levels documented in DEVELOPMENT.md.

**Z-07.** No formal-verified *concrete* `CanonicalForm` for
subgroups of S_n on bitstrings. The GAP implementation uses
`CanonicalImage` (lex-min); the Lean formalization axiomatises
`CanonicalForm` abstractly.

**Z-08.** No side-channel analysis. A Lean formalization proves
mathematical correctness but says nothing about implementation
side channels (cache timing, branch prediction, power analysis).
Real-world deployment must include a side-channel-aware C
implementation + formal proof that the implementation matches
the abstract spec.

**Z-09.** No formal-verified serialization / deserialization of
keys, ciphertexts, nonces. A deployment needs parsing that is
both correct (round-trips) and resistant to malformed input.

**Z-10.** No NIST-PQC competition alignment formally proven.
`docs/HARDNESS_ANALYSIS.md` compares against LESS/MEDS but this
is prose analysis, not Lean theorems.

## 14. Executive Summary

Orbcrypt is a research-stage Lean 4 formalization of a
symmetric-key encryption scheme (HGOE) with an ambitious vision:
formally verify IND-CPA / IND-CCA security via TI-hardness through
a layered per-encoding reduction chain.

**Where the project excels:**
- Zero `sorry`, zero custom axioms across 347+ public
  declarations. CI enforces this stringently.
- Clean architectural layering (GroupAction → Crypto → KEM →
  AEAD → Hardness → PublicKey → Optimization).
- Exemplary documentation posture: every public def has a
  docstring, every module has a `/-! ... -/` header, every
  security-sensitive Prop has a disclosure note.
- Robust Workstream A–N audit cycle has progressively fixed
  naming, shadowing, vacuity, and game-asymmetry issues.
- Mathlib integration is maximal and correct — no ad-hoc
  redefinitions, universe handling post-M1 is polymorphic,
  `SetLike` / `Subgroup` packaging is idiomatic.

**Where the project falls short for v1.0:**
- **Critical: The two-chains problem.** All Scaffolding theorems
  carry vacuous hypotheses on production HGOE; all Quantitative
  theorems are inhabited only at ε = 1 (trivial). No formalized
  scheme instance achieves non-trivial ε < 1 security. This
  matches the reality of Lean+Mathlib formalization state-of-the-
  art (CFI / Grochow–Qiao encoders are research-scope), but the
  release messaging does not consistently disclose this posture
  (findings D1, D2, D3, D4, D6, D7, D8, D9, D10, X-01).
- **High: Production-HGOE incompatibility of `INT_CTXT`.** The
  `authEncrypt_is_int_ctxt` theorem's orbit-cover hypothesis is
  false on production HGOE (`|orbit| < |Bitstring n|`). The
  theorem is thus vacuously applicable to the real scheme.
  Finding I-03.
- **High: Two-phase decryption empirically false.** `TwoPhaseDecomposition`
  fails on the default GAP fallback group. Theorems #24, #25
  labeled Standalone in CLAUDE.md should be Conditional. Finding
  L-03.
- **High: Phase 15 C/C++ implementation not delivered.** The
  GAP reference runs ~4–5 orders of magnitude slower than AES-GCM
  at λ = 128. A v1.0 release absent the C implementation ships a
  formally verified construction that is not practically
  deployable.
- **Medium: 15+ naming / framing issues** where the
  formalization content is correct but the documentation
  overstates or mislabels (D-04, D-07, C-15, E-11, J-03, J-08,
  etc.). These are fixable by docstring edits + renames.

**Release-gate recommendation: DO NOT tag v1.0 from current HEAD.**
See § 13 for the proposed checklist and roadmap. The project
needs a release-messaging reconciliation pass (V1-1 through V1-9)
before a v1.0 tag, focused on aligning external claims with what
the Lean code actually delivers.

**What v1.0 can credibly claim after the checklist:**
- Zero `sorry`, zero custom axioms — machine-checked.
- Formal specification of HGOE + KEM + AEAD architecture.
- Machine-checked correctness for AOE, KEM, AEAD, hybrid, seed,
  nonce paths (rows #1, #4, #9, #10, #12, #13, #17, #18, #24–26
  with corrected status).
- Machine-checked impossibility: invariant attack (row #2),
  equivariant combiner no-go (deterministic form only).
- Mathlib-grade API for code equivalence: Setoid, Subgroup, coset
  identity (rows #21–23).
- Machine-checked universal hash bound for Carter–Wegman (1/p)
  over prime fields — post-L2 upgrade.
- Conditional / scaffolding theorems with explicit ε-bound and
  scheme-instance caveats in release messaging.

**What v1.0 must not claim:**
- "Orbcrypt is IND-1-CPA secure under TI-hardness". The formal
  chain is inhabited only at ε = 1 (trivial).
- "Orbcrypt has machine-checked INT-CTXT security". The proof
  carries an orbit-cover hypothesis false on production HGOE.
- "Orbcrypt's KEM security is formally verified end-to-end". The
  scheme-to-KEM reduction Prop is only inhabited at ε' = 1.
- "Multi-query IND-Q-CPA is proved". The per-step bound is a
  user-supplied hypothesis.

**The project's genuine achievement** is not the scheme's
quantitative security (which requires open research) but the
**infrastructure**: a clean, zero-sorry, zero-custom-axiom Lean
formalization of the security-reduction chain's shape. Future
research work slots cleanly into the per-encoding Props and
surrogate bindings. This is a substantial contribution to the
formal-crypto ecosystem even before non-trivial ε discharges
land.

**Bottom line for v1.0:** fix release messaging (V1-1 through
V1-9), reclassify theorems to match their conditional/scaffolding
status, and ship as "formal specification + infrastructure" — not
as "formally verified cryptosystem". A subsequent v2.0 can make
the stronger claim when research milestones R-01 through R-16
land. Until then, honesty about scope is the best protection for
downstream consumers.
